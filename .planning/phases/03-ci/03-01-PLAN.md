      - `strategy.matrix.os: [ubuntu-latest, macos-latest, windows-latest]`
      - `strategy.matrix.node: [18, 20, 22]`
      - `name: "Test ${{ matrix.os }} / Node ${{ matrix.node }}"`
    - Steps:
      1. `actions/checkout@v6`（per RESEARCH 修正——v4 已弃用）

    **compat-check.yml**（每周兼容性检测，per D-05, D-06）:
    - YAML 顶级字段:
      - `name: "Compat Check"`
      - `on.schedule: [{cron: '0 0 * * 0'}]`（每周日 UTC 0:00）
      - `on.workflow_dispatch: {}`（支持手动触发）
    - 单一 job 名 `compat-check`:
      - `runs-on: ubuntu-latest`（单平台节省资源，per D-07）
      - `timeout-minutes: 10`（per Claude's Discretion）
      - `permissions.issues: write`（创建 Issue 所需，最小权限原则）
    - Steps:
      1. `actions/checkout@v6`
      2. `actions/setup-node@v6`, with: `node-version: 20`, `cache: 'npm'`
      3. `run: npm ci`
      4. **版本检查**（id: version）:
         ```bash
         LATEST=$(npm view ccusage version --json 2>/dev/null || npm view ccusage version)
         echo "latest=$LATEST" >> $GITHUB_OUTPUT
         MAJOR=$(echo "$LATEST" | cut -d. -f1)
         if [ "$MAJOR" = "20" ]; then
           echo "compatible=true" >> $GITHUB_OUTPUT
         else
           echo "compatible=false" >> $GITHUB_OUTPUT
         fi
         ```
         （per RESEARCH 修正——使用 cut -d. -f1 替代不存在的 node:semver）
      5. **兼容版本测试**（if: compatible==true）:
         ```bash
         npm update ccusage
         npm test
         ```
         env: `CCUSAGE_CNY_RATE: '7.0'`
      6. **Major bump 测试**（if: compatible==false, id: test-new-version, continue-on-error: true）:
         ```bash
         npm install ccusage@latest --no-save
         npm test
         ```
         env: `CCUSAGE_CNY_RATE: '7.0'`
         （`--no-save` 仅用于 `npm install`，不用于 `npm ci`，per RESEARCH Pitfall 2）
      7. **创建 Issue**（if: compatible==false && test-new-version.outcome==failure）:
         ```bash
         gh issue create \
           --title "兼容性破坏: ccusage ${{ steps.version.outputs.latest }} 与 ccusage-cn 不兼容" \
           --label "compat-break" \
           --body "..."
         ```
         env: `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}`
          Issue body 包含上游版本号、测试时间、CI 运行链接、建议操作步骤。

    所有 GitHub Actions action 必须使用 `@v6`（not @v4），per RESEARCH 修正。
    不要使用 `node:semver`（Node 无此内置模块），per RESEARCH Pitfall 1。
    `npm install ccusage@latest --no-save` 中 `--no-save` 是合法的，`npm ci` 不接受此参数，per RESEARCH Pitfall 2。

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI workflow YAML → GitHub Actions runner | 工作流配置控制 runner 上的代码执行路径 |
| npm install → npm registry | 从外部注册表下载依赖，存在供应链风险 |
| gh issue create → GitHub API | 使用 GITHUB_TOKEN 写 Issues |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | GITHUB_TOKEN 权限 | mitigate | 最低权限原则：仅 compat-check.yml 声明 `permissions: issues: write`；ci.yml 不声明自定义权限（使用默认 `contents: read`） |
| T-03-02 | Tampering | 上游依赖供应链攻击 | mitigate | `npm ci` 基于 lockfile 固定版本，保证 CI 与开发环境依赖一致；compat-check.yml 对 major bump 仅做测试性升级（`npm install --no-save` 临时安装），不自动更新 package.json 或 lockfile |
| T-03-03 | Tampering | CI 配置文件恶意修改 | mitigate | CI 配置变更需经过 PR review 机制才能合入 main 分支；push trigger 仅对 main 分支 |
| T-03-04 | Spoofing | 工作流手动触发 | accept | `workflow_dispatch` 需要仓库 write 权限，已由 GitHub 内置访问控制管理 |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | RESEARCH.md Package Legitimacy Audit 确认本阶段不引入任何新 npm 包；现有依赖（vitest, publint, ccusage）已通过前期 slopcheck；compat-check.yml 中 `npm install ccusage@latest --no-save` 是 CI 运行时临时操作，不持久化到 lockfile |
</threat_model>
